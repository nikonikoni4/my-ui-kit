## 🎯 核心原则

### 1. 单一职责原则
- 每个组件只负责一个明确的功能
- 复杂功能通过组合多个简单组件实现

### 2. 关注点分离
- **类型定义** → `types.ts`
- **工具函数** → `utils.ts`
- **常量配置** → `constants.ts`
- **自定义 Hooks** → `hooks/`
- **组件实现** → `*.tsx`

### 3. 统一导出入口
- 使用 `index.ts` 作为唯一导出入口
- 外部使用者无需关心内部文件结构
- 便于重构和维护

## 📁 标准目录结构

### 基础结构（适用于简单到中等复杂度组件）

```
frontend/other_project/[组件名]/
├── index.ts              # 统一导出入口 ⭐
├── types.ts              # 类型定义
├── utils.ts              # 工具函数
├── constants.ts          # 常量配置（可选）
├── [ComponentName].tsx   # 主组件
├── [ComponentName]Detailed.tsx  # 变体组件（可选）
└── [ComponentName]Tree.tsx      # 组合组件（可选）
```

### 高级结构（适用于复杂组件或跨项目复用）

```
frontend/other_project/[组件名]/
├── index.ts              # 统一导出入口 ⭐
├── types.ts              # 类型定义
├── constants.ts          # 常量配置
│
├── components/           # 子组件文件夹
│   ├── [ComponentName].tsx
│   ├── [ComponentName]Detailed.tsx
│   └── [ComponentName]Tree.tsx
│
├── hooks/                # 自定义 Hooks
│   ├── useExpandedState.ts
│   └── use[ComponentName]Actions.ts
│

│   ├── formatters.ts
│   └── validators.ts
│
├── styles/               # 样式文件（如果不是 CSS-in-JS）
│   └── [ComponentName].module.css
│
└── __tests__/            # 测试文件
    ├── [ComponentName].test.tsx
    └── utils.test.ts
```

## 📝 文件组织规范

### 1. `index.ts` - 导出入口

**作用**: 统一管理所有导出，隐藏内部实现细节

```typescript
// ✅ 推荐：分类导出，清晰明了
// === Components ===
export { TodoItem, TODO_COLORS, getRandomColor } from './TodoItem';
export { TodoItemTree, useExpandedState } from './TodoItemTree';
export { TodoItemDetailed } from './TodoItemDetailed';
export { TodoItemTreeDetailed } from './TodoItemTreeDetailed';

// === Types ===
export * from './types';

// === Utilities ===
export * from './utils';

// ❌ 避免：直接从子文件夹导出（除非有明确意图）
// export * from './components';  // 这样会导出所有内容，不够精确
```

**使用示例**:
```typescript
// 外部使用时非常简洁
import { TodoItem, TodoItemTree, type TodoItem as TodoItemType } from '@/other_project/todoItem';
```

---

### 2. `types.ts` - 类型定义

**作用**: 集中管理组件相关的所有类型定义

```typescript
/**
 * [组件名] 组件相关类型定义
 * 
 * 设计原则:
 * 1. 类型名称清晰，避免与组件名冲突（如 TodoItem 类型 vs TodoItem 组件）
 * 2. 使用 JSDoc 注释说明复杂类型
 * 3. 导出所有可能被外部使用的类型
 */

// 主要数据类型
export interface TodoItem {
    id: number;
    content: string;
    // ... 其他字段
}

// 组件 Props 类型
export interface TodoItemProps {
    item: TodoItem;
    onUpdate?: (item: TodoItem) => void;
    showEditButton?: boolean;
}

// 工具函数类型
export type TodoItemFilter = (item: TodoItem) => boolean;

// 常量类型
export type TodoState = 'pool' | 'scheduled' | 'completed' | 'shelved';
```

**注意事项**:
- ✅ 类型名称使用 PascalCase
- ✅ 接口优先使用 `interface`，联合类型使用 `type`
- ✅ 导出所有可能被外部使用的类型
- ❌ 避免在组件文件中定义类型（除非是纯内部使用的私有类型）

**注意事项**:

### 4. `constants.ts` - 常量配置

**作用**: 存放组件使用的常量、配置项、枚举值

```typescript

 * TodoItem 颜色预设
```

## 🏷️ 命名约定

### 文件命名

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| 组件文件 | PascalCase | `TodoItem.tsx` |
| 类型文件 | camelCase | `types.ts` |
| 工具文件 | camelCase | `utils.ts` |
| 常量文件 | camelCase | `constants.ts` |
| Hook 文件 | camelCase | `useExpandedState.ts` |
| 样式文件 | camelCase | `todoItem.module.css` |

### 导出命名

```typescript
// ✅ 组件：PascalCase
export const TodoItem: React.FC<TodoItemProps> = ({ ... }) => { ... };

// ✅ Hook：use + PascalCase
export const useExpandedState = () => { ... };

// ✅ 工具函数：camelCase
export const calculateDelayDays = () => { ... };

// ✅ 常量：UPPER_SNAKE_CASE
export const TODO_COLORS = [...];

// ✅ 类型：PascalCase
export interface TodoItem { ... }
export type TodoItemFilter = ...;
```

## ❓ 常见问题

### Q1: 什么时候需要创建 `components/` 子文件夹？

**答**: 当满足以下任一条件时：
- 组件文件超过 5 个
- 有明确的子组件层级关系
- 需要进一步组织内部结构

**示例**:
```
// 简单组件（扁平结构）
todoItem/
├── TodoItem.tsx
├── TodoItemDetailed.tsx
└── index.ts

// 复杂组件（嵌套结构）
calendar/
├── components/
│   ├── Calendar.tsx
│   ├── CalendarHeader.tsx
│   ├── CalendarCell.tsx
│   └── CalendarWeek.tsx
└── index.ts
```

---

### Q2: Hook 应该放在哪里？

**答**: 根据复杂度决定：

| 场景 | 位置 | 示例 |
|------|------|------|
| 简单 Hook（1-2 个） | 组件文件内导出 | `TodoItemTree.tsx` 中的 `useExpandedState` |
| 中等复杂度（3-5 个） | 独立 `hooks.ts` 文件 | `hooks.ts` |
| 复杂（5+ 个） | `hooks/` 文件夹 | `hooks/useExpandedState.ts` |

---

### Q3: 如何处理组件间的类型共享？

**答**: 分层处理：

```
frontend/
├── shared/
│   └── types/
│       └── common.ts          # 全局通用类型
│
└── other_project/
    ├── todoItem/
    │   └── types.ts           # TodoItem 专用类型
    └── taskCalender/
        └── types.ts           # Calendar 专用类型
```

**原则**:
- 只在一个组件使用 → 放在组件的 `types.ts`
- 多个组件使用 → 放在 `shared/types/`
- 跨项目使用 → 考虑发布为独立包

---

### Q4: 常量应该放在哪里？

**答**:

| 常量类型 | 位置 | 示例 |
|----------|------|------|
| 组件专用常量 | `constants.ts` | `TODO_COLORS` |
| 组件内部常量 | 组件文件内 | 私有配置 |
| 全局常量 | `shared/constants/` | API 端点、主题色 |

---

### Q5: 何时需要 `mockData.ts`？

**答**: 在开发阶段需要模拟数据时：

```typescript
// mockData.ts
import type { TodoItem } from './types';

export const MOCK_TODOS: TodoItem[] = [
    { id: 1, content: '示例任务 1', ... },
    { id: 2, content: '示例任务 2', ... },
];
```

**注意**:
- ✅ 仅用于开发和测试
- ✅ 不要在生产代码中导入
- ✅ 可以在 `index.ts` 中有条件导出

```typescript
// index.ts
export * from './types';
export * from './utils';

// 仅在开发环境导出 mock 数据
if (import.meta.env.DEV) {
    export * from './mockData';
}
```

## 🎯 决策流程图

```
开始创建组件
    ↓
是否需要复用？
    ├─ 否 → 放在页面文件夹内
    └─ 是 → 继续
         ↓
    组件复杂度？
         ├─ 简单（1-3 个文件）
         │   └─ 使用扁平结构
         │       └─ [ComponentName].tsx + index.ts
         │
         ├─ 中等（4-8 个文件）
         │   └─ 使用标准结构
         │       └─ index.ts + types.ts + utils.ts + 组件文件
         │
         └─ 复杂（8+ 个文件）
             └─ 使用高级结构
                 └─ components/ + hooks/ + utils/ + ...
```

## 📌 快速检查清单

创建新组件时，确保：

- [ ] 有 `index.ts` 统一导出入口
- [ ] 类型定义在 `types.ts` 中
- [ ] 工具函数在 `utils.ts` 中
- [ ] 组件文件使用 PascalCase 命名
- [ ] 所有导出都经过 `index.ts`
- [ ] 添加了必要的 JSDoc 注释
- [ ] 考虑了变体组件的命名规范
- [ ] 遵循了单一职责原则

---
